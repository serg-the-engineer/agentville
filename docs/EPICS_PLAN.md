# EPICS_PLAN.md

Политика управления эпиками для MVP Zero-Player LLM-Arena.

## Source Of Truth
- Каноническое описание эпиков, зависимости и статус хранятся только в GitHub Issues с label `type:epic`.
- В основном репозитории не хранится полный список/содержание эпиков, чтобы не было дрейфа между кодом и backlog.
- Этот файл хранит только правила планирования и milestone-гейты.

Проверить текущие эпики:
```bash
gh issue list --label type:epic --state open --limit 200
```

## Принципы планирования
- Епики формируют dependency graph, чтобы можно было распараллеливать независимые треки.
- Контракты (DTO, schema, event types) стабилизируются как можно раньше.
- Интеграционные точки фиксируются заранее и проверяются отдельными integration PR.
- Один epic не выполняется "одним большим PR": сначала нарезка на Work Packet-issues.

## Milestone-гейты (по кодам эпиков)
- M1: `E1+E2+E4` завершены, engine работает на in-memory и DB-модели готовы.
- M2: `E3+E5+E6` завершены, возможен e2e run без UI.
- M3: `E7+E8+E9` завершены, рабочий MVP с real-time наблюдением.
- M4: `E10` завершен, подтверждены производительность и надежность.

## Правила декомпозиции эпиков
- Любой epic декомпозируется на Work Packet-issues с префиксом `E*-WP*`.
- Нарезка выполняется в GitHub (issue->issue), без дублирования в markdown-файлах репозитория.
- Для одного epic одновременно открыто не более 3 WP, чтобы не раздувать незавершенную работу.
- Новый WP создается только если зависимости epic не нарушены.

## Правила исполнения
- WIP limit: не более 2 non-epic issues в `status:in-progress` одновременно.
- Приоритет выбора: `bug` > `feature`, затем `priority:p0 -> p3`.
- Если нет задач в работе, сначала берется готовая `status:ready`; если готовых нет, сначала нарезается epic на WP.

## Изменение epic scope
- Scope epic меняется только в самом GitHub issue (редактирование body + комментарий с причиной изменения).
- Если изменился публичный контракт, первым коммитом в PR обновляются схема и contract tests.
