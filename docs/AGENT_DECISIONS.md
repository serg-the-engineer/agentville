# AGENT_DECISIONS

Журнал решений, которые агент принял самостоятельно без дополнительного вопроса, потому что один вариант был явно предпочтительнее.

## Как использовать
- Логировать только решения, влияющие на процесс, архитектуру, статус-модель или способ исполнения.
- Для каждого решения фиксировать контекст, влияние и почему вопрос не был вынесен человеку.
- Если варианты сопоставимы по риску или решение необратимо, агент должен остановиться и дождаться человека вместо самостоятельного выбора.

## Шкала критичности
- `High`: может изменить delivery policy, merge governance, продуктовые границы или привести к значимому простою/регрессии.
- `Medium`: влияет на операционный процесс и инструменты, но обратимо без сложной миграции.
- `Low`: локальное оформление, структура документации, технический дефолт без существенного риска.

## Правило самостоятельного выбора
Агент принимает решение сам, если одновременно:
- один вариант лучше согласуется с уже существующими документами и кодом;
- решение обратимо;
- последствия понятны и локальны;
- отсутствие решения прямо тормозит работу.

Агент блокируется и ждет человека, если:
- меняется продуктовый контракт;
- нет надежного машинного сигнала, что ожидание человека снято;
- есть несколько равноценных вариантов с разной продуктовой ценой;
- решение необратимо или затрагивает governance/безопасность.

## Решения

### 2026-02-27: Ввести отдельный статус `Needs Human`
- Критичность: `High`
- Контекст: исходный flow использовал `Blocked` и свободные комментарии для ожидания решения человека, но это смешивает "ждем решения" с "есть проблема исполнения".
- Решение: добавить отдельный статус `Needs Human` для случаев, когда следующий ход за человеком.
- Почему без уточнения: это явно лучше согласуется с заявленным требованием не использовать `Blocked` для обычного ожидания аппрува и делает автоматику надежнее.
- Последствия: нужно синхронизировать Project fields, labels и документы; зато flow перестает застревать на неоднозначной семантике статусов.

### 2026-02-27: Использовать label `approval:granted` как machine-readable сигнал утверждения epic
- Критичность: `High`
- Контекст: approval эпика предлагалось выражать либо статусом, либо комментариями, но комментарии плохо автоматизируются.
- Решение: отделить approval от статуса через отдельный label `approval:granted`.
- Почему без уточнения: статус должен отвечать за владельца следующего хода, а approval - за разрешение на реализацию; разделение обязанностей снимает конфликт.
- Последствия: automation может надежно фильтровать executable задачи; человеку нужен один дополнительный явный сигнал.

### 2026-02-27: Зафиксировать базовую ветку как `main`
- Критичность: `Medium`
- Контекст: в `docs/GIT_WORKFLOW.md` была указана базовая ветка `master`, но локальный репозиторий сейчас находится на ветке `main`.
- Решение: в новом flow опираться на `main` как на текущую базовую ветку.
- Почему без уточнения: следование фактическому состоянию репозитория безопаснее, чем проектирование automation против несуществующей/неиспользуемой ветки.
- Последствия: связанные документы и git-команды нужно держать консистентными, иначе ребейз/merge automation будет ломаться.

### 2026-02-27: Один запуск automation делает одну логическую единицу работы
- Критичность: `Medium`
- Контекст: пользователь описал периодический запуск "каждые X минут", но не зафиксировал, должен ли один запуск разгребать всю очередь.
- Решение: один запуск выбирает и выполняет только один следующий шаг, затем завершается.
- Почему без уточнения: такой режим проще сделать идемпотентным, безопаснее для WIP-лимитов и легче отлаживать.
- Последствия: throughput ниже, чем у batch-run, но поведение предсказуемее и меньше риск гонок.
