# AGENT_DELIVERY_FRAMEWORK.md

Рабочий фреймворк для передачи епиков отдельным агентам и параллельной разработки.
Основа: [Harness Engineering](https://openai.com/index/harness-engineering/) — агент-легибельность, короткие PR, контракты как код, регулярная hygiene.

## 1) Цели фреймворка
- Безопасно распараллеливать работу по епикам.
- Уменьшать merge-конфликты через контрактные границы.
- Делать прогресс измеримым и воспроизводимым.

## 2) Базовые принципы
- Agent-legible repo: короткий `AGENTS.md` + специализированные docs по зонам.
- Правила в коде, не в prose: тесты, схемы, типы, CI-gates.
- Изменения малыми порциями: stack PR вместо больших батчей.
- Постоянная repo hygiene: чистка веток, TODO-ревизия, удаление dead code.

## 3) Единица делегирования: Work Packet
Каждому агенту выдается Work Packet:
- `WP-ID`: уникальный идентификатор (`E4-WP2` и т.п.).
- `Objective`: измеримая цель.
- `Scope`: что входит/не входит.
- `Touched Contracts`: какие schema/API/event меняются.
- `Tests First`: список тестов, которые агент обязан написать до кода.
- `Deliverables`: файлы/модули/миграции.
- `Validation Commands`: команды проверки.
- `Exit Criteria`: условия приемки.

## 4) Последовательность выполнения агента
1. Прочитать `AGENTS.md` + релевантные docs.
2. Зафиксировать тесты (RED).
3. Реализовать минимальный код (GREEN).
4. Рефакторинг и стабилизация (REFACTOR).
5. Обновить документацию при изменении контрактов.
6. Подготовить PR с обязательным шаблоном.

## 5) Контрактные границы для параллелизма
- Любой изменяемый публичный контракт выносится в отдельный "contract PR".
- Пока contract PR не merged, остальные агенты не меняют тот же контракт.
- Внутренние реализации могут идти параллельно при фиксированном интерфейсе.

## 6) Матрица распараллеливания (MVP)
| Поток | Основные епики | Что можно делать параллельно |
|---|---|---|
| P1 Contracts/Data | E1, E2 | Блок contracts + миграции |
| P2 Engine | E4 | Параллельно с E2 после freeze E1 |
| P3 Match Setup | E3 | Независимо от E4 при фиксированных моделях |
| P4 LLM | E5 | Параллельно с API подготовкой |
| P5 Platform | E6, E7, E8 | После готовности E2/E4/E5 |
| P6 UX/Perf | E9, E10 | После стабильного API/WS |

## 7) Интеграционные точки
- Integration PR-1: `E2 + E4`.
- Integration PR-2: `E3 + E6`.
- Integration PR-3: `E5 + E7 + E8`.
- Release Candidate PR: `E9 + E10`.

## 8) Формат handoff между агентами
Каждый handoff обязан содержать:
- Что сделано.
- Что не сделано.
- Какие тесты добавлены.
- Как локально проверить (команды).
- Риски и открытые вопросы.

## 9) Координация и конфликт-менеджмент
- Если два Work Packet меняют один и тот же модуль, назначается owner модуля на итерацию.
- Merge конфликты решаются owner-ом целевого епика, не автором соседнего PR.
- При конфликте контрактов приоритет у contract PR.

## 10) Минимальный операционный ритм
- Ежедневно: статус по каждому WP (`Done` / `In Progress` / `In Review` / `Needs Human` / `Blocked`).
- Каждые 2 дня: интеграционный sync по зависимостям.
- Еженедельно: hygiene-сессия (stale branches, TODO debt, docs drift).
