# QUALITY_AND_DOD.md

Единый стандарт качества, TDD и готовности фич.

## 1) Обязательная инженерная политика
- TDD обязателен для любой игровой, API и интеграционной логики.
- Тесты пишутся до реализации.
- "Код без теста" не считается завершенной работой.

## 2) TDD Workflow (RED -> GREEN -> REFACTOR)
1. RED: написать тест, который падает на новой/исправляемой логике.
2. GREEN: реализовать минимальный рабочий код, чтобы тест прошел.
3. REFACTOR: улучшить структуру, не меняя поведение.
4. RE-RUN: прогнать полный релевантный набор тестов.

## 3) Тестовая пирамида для проекта
- Unit: правила движка, формулы, эффекты, сортировка действий, валидация контрактов.
- Integration: room lifecycle + runner + DB + LLM fallback.
- E2E (MVP): create/join/ready/start/finish/results + WS public logs.

## 4) Минимальные quality gates
- Все новые правила покрыты unit-тестами.
- Все пользовательские флоу покрыты минимум одним integration-тестом.
- Нет падающих тестов в CI.
- Нет неиспользуемого кода и мертвых флагов после merge.
- Документация синхронизирована с контрактами.

## 5) Definition of Ready (DoR) для фичи
Фича может стартовать только если есть:
- Четкая формулировка цели и бизнес-правила.
- Acceptance criteria в проверяемом виде.
- Ясные границы in-scope/out-of-scope.
- Понимание контрактов (event/api/schema), которые затрагиваются.
- План тестов (unit + integration, где применимо).

## 6) Definition of Done (DoD) для фичи
Фича завершена только если:
- Реализована функциональность по acceptance criteria.
- Написаны тесты в стиле TDD и они проходят.
- Обновлены `PRD.md`, `docs/ARCHITECTURE.md` или локальные docs при изменении правил.
- PR получил review и все блокеры закрыты.
- Есть краткий changelog и риски регрессии явно описаны.

## 7) Definition of Done для епика
Епик завершен если:
- Все входящие фичи закрыты по DoD.
- Есть интеграционный сценарий, подтверждающий end-to-end поведение.
- Метрики/наблюдаемость для епика подключены (если это runtime-компонент).
- Нет открытых P0/P1 дефектов по области епика.

## 8) Классификация дефектов
- P0: ломает основной игровой флоу или приводит к потере данных.
- P1: критично искажает результаты игры/логики, есть обходной путь.
- P2: частичное нарушение поведения без блокировки основного флоу.
- P3: косметика, DX, не критично для корректности.

## 9) Нулевая толерантность
- Merge без тестов для новой логики.
- Merge при несоответствии кода и зафиксированных правил.
- Невоспроизводимые изменения без команд запуска/проверки.
