# AUTONOMOUS_DELIVERY_FLOW

Явный цикл работы automation, которая стартует каждые X минут, делает не больше одной логической единицы работы и завершается.

## 1. Цели
- Максимально автономный delivery без постоянного ожидания человека.
- Идемпотентный запуск: повторный прогон не должен дублировать действия.
- Один запуск = один следующий шаг очереди, а не "сделать все сразу".
- `Blocked` используется только для реальных проблем исполнения; ожидание решения человека - отдельное состояние.

## 2. Источники истины
- GitHub PRs и Issues.
- GitHub Project status.
- Явные labels, а не парсинг свободного текста как единственный сигнал.
- Базовая ветка репозитория: `main`.

## 3. Модель состояний
Статус задачи/эпика в Project:
- `Inbox`: еще не разобрано.
- `Ready`: следующий ход должен делать агент.
- `In Progress`: агент сейчас выполняет работу.
- `In Review`: открыт PR и идет review/fix cycle.
- `Needs Human`: следующий ход должен сделать человек.
- `Blocked`: агент не может продолжать из-за конфликта, сбоя, внешней зависимости.
- `Done`: работа завершена.

Дополнительный label:
- `approval:granted`: человек утвердил epic-план, и задачи из этого эпика можно брать в реализацию.

Почему так:
- `Status` отвечает на вопрос "кто делает следующий ход".
- `approval:granted` отвечает на вопрос "можно ли реализовывать epic/task по этому направлению".
- Это лучше, чем перегружать `Blocked` или пытаться интерпретировать комментарии без явного машинного сигнала.

## 4. Запуск automation
Каждый запуск делает последовательность:
1. Синхронизировать локальный `main` и состояние GitHub.
2. Считать open PR, open issues, project statuses, labels и последние human comments.
3. Классифицировать очередь по типам работы.
4. Выполнить первый подходящий шаг из приоритетного списка.
5. Записать результат, обновить статусы/labels/PR и завершиться.

## 5. Приоритеты очереди
Порядок всегда такой:
1. PR, ожидающие review.
2. PR, где уже запрошены исправления или пришли новые human comments.
3. Items в `Needs Human`, по которым человек уже ответил и снова сделал их `Ready`.
4. Новая работа из backlog при соблюдении WIP-лимитов.
5. Если ничего не найдено, запуск завершается без действий.

## 6. Шаг 1. Review PR
Если есть PR, ожидающие review:
1. Выбрать один PR по приоритету (`P0 -> P3`), затем по возрасту.
2. Провести review строго в порядке:
   - код делает то, что требует задача;
   - тесты есть, они релевантные и не пустые;
   - качество кода, безопасность, обработка ошибок;
   - возможные упрощения и удаление лишней сложности;
   - документация.
3. Если есть замечания:
   - оставить комментарии;
   - выставить `changes requested`;
   - связанная задача остается не `Done`;
   - запуск завершается.
4. Если замечаний нет:
   - проверить [HUMAN_GATE_POLICY.md](/Users/s.retivykh/sandbox/agentville/docs/automation/HUMAN_GATE_POLICY.md);
   - если нужен человек, перевести PR или связанную задачу в `Needs Human`, написать краткое summary и завершиться;
   - если человек не нужен, CI зеленый, PR rebased на `main`, нет конфликтов и связанная задача определяется однозначно, то merge PR, удалить ветку/worktree, перевести связанную задачу в `Done` и завершиться.

## 7. Шаг 2. Исправление PR
Если PR на review нет, но есть PR на доработку:
1. Выбрать один PR по приоритету.
2. В dedicated worktree подтянуть актуальный `main`.
3. Внести исправления через TDD, прогнать quality gates.
4. Обновить PR, оставить summary изменений.
5. Вернуть задачу в `In Review`.
6. Завершить запуск.

## 8. Шаг 3. WIP-лимиты
Новая работа берется только если одновременно:
- `In Progress` epics не больше 1.
- уже открытых `In Progress` задач меньше 2.

Если лимиты превышены, automation завершает запуск без новых задач.

`Needs Human` и `In Review` не считаются `In Progress`, потому что следующий ход там не за агентом.

## 9. Шаг 4. Выбор новой работы
Если PR-очередь пуста и WIP в пределах лимитов:

### 9.1 Если есть epic в `In Progress`
- Если есть `Ready` задача с приоритетом выше, чем у этого epic, берем ее.
- Иначе берем `Ready` задачу из активного epic.
- Если внутри epic нет ни одной actionable-задачи, запуск завершается.

### 9.2 Если нет epic в `In Progress`
- Берем highest-priority actionable item.
- Если это epic без `approval:granted`, агент делает только проектирование.
- Если это epic с `approval:granted`, его можно реализовывать целиком как обычную задачу, но только если у epic нет ready child tasks.
- Если это обычная задача, ее можно брать в реализацию только когда parent epic уже имеет `approval:granted`.

## 10. Epic lifecycle
Epic проходит через явные стадии:
1. `Ready`: epic готов к проработке.
2. `In Progress`: агент делает проектирование.
3. Агент публикует в issue:
   - solution outline;
   - тестовую стратегию;
   - документацию/контракты;
   - архитектурные решения;
   - риски и open questions.
4. После публикации планов epic переводится в `Needs Human`.
5. Человек:
   - если согласен: добавляет `approval:granted` и переводит epic в `Ready`;
   - если нужны правки: оставляет один комментарий с правками и возвращает epic в `Ready` без `approval:granted`.
6. Только после `approval:granted` дочерние задачи или сам epic становятся executable.

## 11. Обычная задача
Для обычной задачи агент перед кодом обязан:
1. Собрать краткий план реализации в issue/comment.
2. Если вопросов нет и human gate не нужен, сразу перейти к TDD-реализации.
3. Если вопросы есть, собрать их одним сообщением, перевести issue в `Needs Human` и завершить запуск.

`Blocked` для таких задач не используется, если проблема только в необходимости решения человека.

## 12. Когда flow может застрять
### 12.1 Self-review и branch protection
Если PR создан той же identity, которая должна его approve, а branch protection требует внешний approval, полного autonomous merge не получится. В этом случае item уходит в `Needs Human`.

### 12.2 Нет однозначной связи PR -> issue
Если PR не содержит `Closes #...` или связь неоднозначна, PR не auto-merge'ится даже при чистом review. Это human gate.

### 12.3 Комментарии человека не переведены обратно в `Ready`
Automation не должна бесконечно парсить свободный текст и гадать, что человек имел в виду. Минимально надежный путь: человек после ответа возвращает status в `Ready`. Позже это можно автоматизировать отдельным responder'ом.

### 12.4 Task под неаппрувнутым epic
Такая задача считается non-actionable и пропускается.

### 12.5 Approved epic и child tasks одновременно
Чтобы не дублировать план, automation по умолчанию предпочитает ready child task. Сам epic реализуется целиком только когда child tasks не заведены.

## 13. Идиоматичный вариант
На практике самый устойчивый вариант такой:
- один cron-like запуск = один следующий шаг;
- `Status` описывает владельца следующего хода;
- `approval:granted` - отдельный durable gate;
- PR merge разрешен только по policy, а не "если ревью чистое, то всегда merge";
- free-form comments остаются для контекста, но не являются единственным API для автоматики.

Селектор принимает snapshot очереди и возвращает одно действие:
- `review_pr`
- `fix_pr`
- `plan_epic`
- `implement_task`
- `implement_epic`
- `stop`
