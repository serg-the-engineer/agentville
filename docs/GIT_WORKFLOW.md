# GIT_WORKFLOW.md

Правила работы с git для командной и агентной разработки.
Основа: подходы из [Harness Engineering](https://openai.com/index/harness-engineering/) — маленькие PR, высокосигнальные изменения, автоматизированные проверки.

## 1) Ветвление
- Базовая ветка: `master`.
- Рабочие ветки: `codex/<epic-id>-<short-scope>`.
- Запрещены длинноживущие ветки без синхронизации более 2 рабочих дней.
- Для параллельной разработки: одна рабочая ветка должна жить в отдельном `git worktree`.
- Создание и обслуживание `worktree` (sync/cleanup) выполняются агентами и automations, а не вручную владельцем репозитория.

## 2) Размер и тип PR
- Один PR = одна логическая задача.
- Целевой размер: до ~400 строк production-кода (тесты не ограничиваются жестко).
- Крупные изменения разбивать на stacked PR:
  1. контракты/типы,
  2. реализация,
  3. интеграция.

## 3) Коммиты
- Формат: `type(scope): summary`.
- Допустимые `type`: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`.
- Каждый коммит должен быть компилируемым и проходить локальные тесты для измененного модуля.

## 4) TDD и коммит-последовательность
- Обязательная последовательность:
  1. `test:` коммит с падающим тестом (RED).
  2. `feat/fix:` минимальная реализация (GREEN).
  3. `refactor:` улучшение без изменения поведения.
- Исключения только для чисто инфраструктурных/документационных PR.

## 5) Обязательные проверки перед PR
- `uv run --with pytest pytest -q`
- Проверка, что изменены/добавлены тесты на новую функциональность.
- Обновлены документы, если менялись контракты или процесс.

## 6) PR-шаблон (обязательно)
- Problem: что меняется и почему.
- Scope: границы PR (что намеренно не входит).
- Test Plan: какие тесты добавлены и что запускалось.
- Risk: риски регрессии и как откатить.
- Follow-ups: что остается на следующие PR.

## 7) Review protocol
- Reviewer проверяет в порядке:
  1. корректность поведения,
  2. тестовое покрытие,
  3. контрактные изменения,
  4. читаемость и поддерживаемость.
- Комментарии должны быть action-oriented и воспроизводимыми.
- Блокирующие замечания обязательно сопровождаются конкретным примером исправления.

## 8) Merge policy
- Мердж только при зеленых обязательных проверках.
- Предпочтительно `squash merge` для единичных PR, `merge commit` для stacked цепочек.
- После merge: удалить ветку.

## 9) Repo hygiene (еженедельно)
- Удаление stale-веток.
- Ревизия `TODO/FIXME` и dead code.
- Проверка несинхронизированных контрактов в документации и коде.

## 10) Worktree policy для параллельной разработки (исполняется агентами/automations)
- Один `worktree` = одна активная задача (issue/WP) и одна ветка `codex/*`.
- Основная директория репозитория (`master`) используется только для синхронизации и интеграции, не для feature-разработки.
- Рекомендуемая структура: хранить все worktree в соседней папке, например `../agentville-worktrees/`.
- Нельзя использовать одну и ту же ветку одновременно в двух worktree.

### 10.1 Старт задачи (агент)
```bash
git fetch origin
git worktree add ../agentville-worktrees/codex-e4-slot-rules \
  -b codex/e4-slot-rules origin/master
```
- Команду запускает агент по команде владельца "стартуй задачу <scope>".
- Владелец репозитория не выполняет эти команды вручную в штатном режиме.

### 10.2 Ежедневная синхронизация (automation)
Внутри каждого активного worktree выполняется:
```bash
git fetch origin
git rebase origin/master
```
- Синхронизировать не реже 1 раза в рабочий день.
- Если есть конфликт, сначала решить конфликт в текущем worktree, затем прогнать тесты.
- При конфликте automation не продолжает rebase и создает отчет со статусом `blocked`.

### 10.3 Завершение после merge (automation/агент)
```bash
git worktree remove ../agentville-worktrees/codex-e4-slot-rules
git branch -d codex/e4-slot-rules
git worktree prune
```
- Удалять worktree и локальную ветку сразу после merge PR.
- Если PR закрыт без merge, ветка/worktree тоже должны быть удалены.

## 11) Что требуется от владельца репозитория (минимум)
1. Создать/приоритизировать задачу (issue/WP) и дать scope.
2. Дать агенту команду на старт: создать ветку `codex/*`, worktree и PR.
3. Провести продуктовый review, принять решение о merge/закрытии PR.

## 12) Что делается автоматически без участия владельца
1. Создание отдельного `worktree` под задачу.
2. Ежедневный `fetch/rebase` активных чистых worktree.
3. Отчет о blocked-ветках при конфликтах.
4. Удаление merged/closed worktree и локальных веток.
5. `git worktree prune` и weekly hygiene stale-артефактов.
