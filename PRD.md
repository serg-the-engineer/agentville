# PRD: Zero-Player LLM-Arena (MVP)

## 1. Цель продукта
Собрать веб-сервис с комнатами, где пользователи до старта матча задают стартовый промпт своему LLM-агенту, отмечаются `ready`, и создатель комнаты запускает игру.

После старта игра идет полностью автономно:
- без вмешательства пользователей;
- backend является единственным источником истины;
- backend валидирует действия агентов, бросает D10, применяет эффекты и формирует логи.

Победитель определяется по суммарной ценности предметов в конце игры.

## 2. Product Scope (MVP)
В MVP входят:
- Лобби/комнаты: создание, join, настройка агента, ready, start.
- Игровой раннер: ходы, проверки, эффекты, инвентари, пол, логи.
- Интеграция с LLM (vLLM): получение JSON-действий и `memory_file`.
- Публичный просмотр матча в реальном времени (WS public logs).
- Админский просмотр истории игр, событий и логов.

Не входят в MVP:
- Любое изменение промпта после старта.
- Сложный UI торговли/переговоров.
- Личные цели/квесты.
- Экономика вне одной комнаты.

## 3. Зафиксированные решения (без противоречий)
1. После старта матча нет вмешательства пользователя.
2. Алерты про плохой промпт/память - только телеметрия на будущее, не игровая механика.
3. Исполнение действий идет по слотам (`slot 1` всех агентов, затем `slot 2`, ...), внутри слота - фиксированные приоритеты.
4. `steal` считается агрессивным действием и резолвится как атака, но через `attentiveness`.
5. `stunned` уменьшает количество действий в следующем раунде на 1 (`actions_penalty_next_round`), а не "пропуск хода" и не `agency=0`.
6. Structured/guided output снижает ошибки формата, но не считается 100% гарантией корректного действия.
7. Private/conditional логи адресуются через `recipient_agent_id`.
8. Победа: сумма `price` предметов у агента в конце игры.
9. На каждую игру генерируется отдельный пул поисковых слотов комнаты с фиксируемым количеством слотов и распределением категорий предметов.
10. `agency` фиксирована и равна `2` у всех агентов; игрок не может ее менять.
11. Стартовые очки статов случайно распределяются между `strength` и `attentiveness` из одинакового пула; по умолчанию агент получает 1 случайную слабость.
12. Игрок может опционально взять `+1` к `strength` или `attentiveness`, но получает вторую случайную слабость.
13. Для временного бафа по предмету используется режим `next_attack` (ровно одна атака); оружие может иметь постоянный эффект (`permanent`).

## 4. Игровые правила (MVP)

### 4.1 Статы
- `strength`
- `attentiveness`
- `agency`

Предметы/эффекты могут давать простые модификаторы `+/-N` к статам.

### 4.2 Проверка D10
Любая проверка:
- `D10 + stat + mods` vs `D10 + stat + mods`
- При ничьей побеждает защита/цель.

### 4.3 Действия
Минимальный набор:
- `defend`
- `attack(target_agent_id, item_id?)`
- `steal(target_agent_id, item_id?)`
- `search`
- `inspect_item(item_id)`
- `activate_item(item_id)`
- `drop_item(item_id)`
- `pickup_item(item_id)`
- `broadcast(text)`

Правила по действиям:
- `defend`: до конца текущего раунда `+1 strength`, `+1 attentiveness`.
- `attack`: `D10 + attacker.str` vs `D10 + target.str`; при успехе атакующий забирает 1 предмет у цели; при разнице `>=5` дополнительно накладывается `stunned(-1 action next round)`.
- `steal`: `D10 + thief.att` vs `D10 + target.att`; при успехе забирается 1 предмет; само действие не публичное (см. логи).
- `search`: сервер бросает `D10` и применяет модификатор поиска (`attentiveness + mods`) к броску.
  По итоговому `search_total` открываются поисковые слоты:
  - `<=6`: не открывает слоты;
  - `7..9`: открывает 1 слот;
  - `>=10`: открывает 2 слота.
  Если в открытом слоте есть предмет - агент получает его, иначе слот пустой; при переполнении инвентаря предмет падает на пол.
- `inspect_item`: приватно раскрывает `activation_desc` и `is_noticeable`.
- `activate_item`: применяет `effect_type` (MVP: `stat_buff`, `stat_debuff`, `stun_on_hit` опционально), может расходовать предмет.
- `drop_item`: предмет на пол (`owner = NULL`), публичный лог.
- `pickup_item`: подъем предмета с пола, если есть слот; иначе no-op + приватный лог.
- `broadcast`: публичное сообщение без прямого механического эффекта.

### 4.4 Оглушение (`stunned`)
- Хранится как `actions_penalty_next_round=1`, `duration_rounds=1`.
- На старте раунда вычисляется `effective_agency = max(0, base_agency - penalties)`.
- Если `effective_agency = 0`, агент все равно получает контекст, но должен вернуть `actions=[]`; если вернул действия, движок их принудительно игнорирует.

### 4.5 Порядок исполнения
Для `slot_index = 1..max_effective_agency_among_agents`:
1. Берется i-е действие каждого агента (если есть).
2. Список сортируется по `action_priority`.
3. Тайбрейк - детерминированный (например, `agent_id ASC` или заранее рассчитанная инициатива от `rng_seed`).
4. Действия исполняются, пишутся `game_events` и `game_logs`.

Приоритеты действий (MVP):
1. `defend`
2. `drop_item`, `pickup_item`, `broadcast`
3. `attack`, `steal`
4. `search`, `inspect_item`, `activate_item`

### 4.6 Visibility и логи
Факт (истина) всегда хранится как структурированный `game_event`.
Текстовые логи - производные:
- `public`: видят все агенты и зрители.
- `private(recipient_agent_id)`: видит только конкретный агент.
- `conditional(recipient_agent_id)`: видит только агент, прошедший проверку восприятия.

MVP perception для `steal`:
- Для каждого наблюдателя (`all except thief + target`) бросок:
- `D10 + observer.att` vs `threshold` (дефолт 8)
- Успех: условный лог "вы заметили ...".

### 4.7 Поисковые слоты и генерация предметов
- Перед стартом матча движок генерирует `search_slots` комнаты.
- `search_slots` содержит фиксированное количество ячеек, часть с предметами, часть пустые.
- Дефолт распределения MVP:
  - `20` слотов в комнате;
  - `5` безделушек (`trinket`);
  - `3` оружия (`weapon`);
  - `1` особый предмет (`special`);
  - остальные слоты пустые.
- В одном слоте может быть максимум один предмет.
- Слот открывается только один раз за матч.

### 4.8 Стартовые статы, слабости и agency
- На старте матча каждый агент получает одинаковый пул очков статов (MVP default: `2`), который случайно распределяется между `strength` и `attentiveness` в любой комбинации (`0/2`, `1/1`, `2/0`).
- Каждый агент получает 1 случайную слабость (постоянный дебаф).
- Игрок может опционально выбрать `+1` к `strength` или `attentiveness`; за это агент получает дополнительную случайную слабость.
- `agency` у всех фиксирована `2` и не меняется пользовательскими настройками.

## 5. LLM-контракт и устойчивость

### 5.1 Вход в LLM на ход
- Видимые агенту логи с его предыдущего хода (`public` + его `private/conditional`).
- Инвентарь агента: только `item_id`, `obfuscated_name`, `appearance`.
- Предметы на полу: только room-scoped публичные идентификаторы и обфусцированные атрибуты.
- Снимок статов/эффектов на текущий раунд.
- `memory_file` (ограниченный по бюджету).
- `allowed_actions` комнаты и строгая JSON-схема ответа.

### 5.2 Выход из LLM
- `actions[]` длиной `<= effective_agency`
- `memory_file`

### 5.3 Валидация и fallback
- Невалидный JSON/схема/ID/невыполнимые preconditions -> действие становится `no-op` + приватный лог ошибки.
- Таймаут/ошибка инференса -> `actions=[]`, `memory_file` не обновляется, приватный лог + `game_event: llm_error`.
- Structured output используется, но не считается абсолютной гарантией корректности.

## 6. Управление памятью
Серверные лимиты:
- `max_base_prompt_chars` (дефолт: `6000`) - иначе `ready` отклоняется.
- `max_memory_chars_sent` (дефолт: `4000`) - сервер всегда режет память перед запросом к LLM.

В БД хранятся:
- `memory_file_raw` (как вернул агент)
- `memory_file_trimmed` (что реально ушло в следующий вызов)

MVP trimming:
- детерминированный `take_last_n_chars`
- с системным префиксом `Memory was truncated`.

Если LLM вернул `context length exceeded`:
1. один ретрай с `max_memory_chars_sent / 2`
2. при повторной ошибке - пропуск действий на ход (`actions=[]`).

## 7. Данные и идентификаторы
Ключевое требование:
- Наружу в API и в LLM-контекст выдаются только room-scoped публичные `agent_id/item_id` (UUID).
- Внешним потребителям не выдаются `template_id` и внутренние системные ключи шаблонов.

## 8. API и Real-Time (MVP)
REST:
- `POST /rooms`
- `POST /rooms/{id}/join`
- `POST /rooms/{id}/ready`
- `POST /rooms/{id}/start`
- `POST /rooms/{id}/agent` (создание/изменение имени и промпта до старта)
- `GET /rooms/{id}` (публичное состояние)
- `GET /rooms/{id}/results`
- `GET /admin/rooms`
- `GET /admin/rooms/{id}`

WS:
- `WS /ws/rooms/{id}/public_logs`
- опционально: `WS /ws/rooms/{id}/private_logs?agent_id=...` (доступ только владельцу агента или админу)

## 9. UX-объем MVP
- Lobby: создание комнаты, список агентов, `ready`, старт создателем.
- Game view: поток публичных логов, текущий ход, финальные результаты.
- Admin view: список игр, фильтры, timeline событий/логов, экспорт промптов и метрик (без утечки чужих секретов в публичные каналы).

## 10. Acceptance Criteria

Юнит-тесты движка:
- Порядок `slot 1 -> priorities -> slot 2`.
- Формулы `attack/steal` и правило ничьи.
- `stunned` уменьшает `effective_agency` ровно на 1 в следующем раунде.
- Conditional logs для `steal` корректно зависят от perception-броска.
- Инвентарь/пол: overflow -> floor; `pickup` уважает лимит слотов.

Интеграционные:
- Сценарий: `create room -> join -> ready -> start -> finished -> results`.
- WS-клиент получает публичные логи в ходе игры.

Надежность:
- `invalid JSON`, `timeout`, `context length exceeded` не роняют игру; ход деградирует строго по fallback-правилам.

## 11. Milestones
1. Зафиксировать enums/схемы: action types, priorities, JSON schema, event types.
2. Поднять PostgreSQL-схему, миграции и модели.
3. Реализовать чистый движок (без LLM): состояние, RNG, исполнение, events/logs.
4. Подключить vLLM-адаптер + structured output + fallback + memory trimming.
5. Собрать REST lifecycle + runner + WS-трансляцию.
6. Добавить минимальный frontend (lobby + просмотр логов) и минимальную админку.
7. Прогнать нагрузку `10 агентов x 12 ходов`, профилировать узкие места.

## 12. Defaults
- `max_turns = 12`
- `inventory_slots = 4`
- `agency = 2` (фиксировано для всех агентов)
- стартовый пул статов на агента: `2` очка между `strength` и `attentiveness` в любой комбинации
- слабости: `1` случайная по умолчанию, `+1` случайная при опциональном бонусе `+1` к стату
- поисковые слоты комнаты: `20` (`5 trinket`, `3 weapon`, `1 special`, остальное пусто)
- Стартовые предметы: `0..1` на агента
- Цена предметов скрыта до финального скоринга
- `inspect` раскрывает эффект активации и заметность, но не цену
